二、技术架构设计（精简且完整）

2.1 整体架构

采用极简分层架构，聚焦12步核心逻辑，降低开发复杂度、提升开发效率，确保快速落地，整体分为3层，各层职责清晰、低耦合，衔接顺畅：

- 前端交互层：负责用户可视化操作，接收用户输入（目录选择、规则设置、API Key配置等），展示处理进度、反馈信息，提供全流程操作入口，界面极简且引导清晰。

- 核心功能层：对应12步核心流程，拆解为12个功能模块，每个模块独立实现对应功能，通过调用基础依赖层的组件，完成文件整理、字幕转写、翻译等核心操作。

- 基础依赖层：包含所有核心依赖组件，负责模型部署、API对接、系统权限适配、文件操作等，是整个软件的运行基础，所有依赖均内置打包，无需用户额外配置。

2.2 核心技术栈（MVP精简版，明确版本与用途）

- 开发语言：Python 3.8~3.10（64位），兼顾开发效率和兼容性，适配Windows系统，降低前后端开发衔接成本。

- 前端UI：PyQt5，用于开发极简桌面端界面，支持按钮、输入框、进度条、表格等核心组件，无需复杂动画，聚焦操作便捷性。

- 核心功能组件：
        

  - faster-whisper：Whisper模型优化部署，提升字幕转写速度，适配批量处理场景。

  - requests：用于对接DeepSeek AI翻译API，实现AI翻译功能，处理API请求与响应。

  - CSANMT：本地翻译模型，轻量无冗余，无需联网即可完成翻译，适配MVP快速落地需求。

  - os/shutil库：实现目录扫描、文件复制/移动、路径校验等文件操作，保障目录整理功能正常运行。

- 打包工具：PyInstaller，将软件打包为单一exe安装包，支持一键安装、卸载，无残留文件，体积控制在300MB以内（含内置模型）。

三、MVP核心功能技术实现（严格对应12步核心逻辑，补充关键细节）

严格遵循12步核心逻辑，每步保留必要实现细节、异常处理和逻辑衔接，剔除冗余校验和非必要功能，确保流程连贯、功能可用，贴合MVP快速落地需求。

3.1 步骤1：选择目录（核心基础步骤）

- 界面设计：2个独立“浏览”按钮，分别对应“源音频目录”和“输出根目录”，按钮下方设置文本框，实时显示选择的目录路径；两个目录均选择完成后，“下一步”按钮自动激活，未选择时呈灰色不可点击。

- 技术实现：调用os.path.exists()方法校验目录是否存在，调用os.access()方法校验目录读写权限（源目录需读权限，输出目录需写权限）；筛选源目录中后缀为.mp3的文件，仅保留MP3格式文件进行后续处理。

- 异常处理：目录不存在、无读写权限时，弹出简洁提示（如“所选目录不存在，请重新选择”“目录无读写权限，请调整权限或更换目录”）；源目录无MP3文件时，提示“未找到MP3文件，请重新选择源目录”，阻止进入下一步。

3.2 步骤2：选择设置文件名匹配规则（筛选目标音频）

- 界面设计：3种匹配规则（关键词匹配、前缀匹配、后缀匹配）单选按钮，对应规则的输入框（关键词匹配支持多关键词，用逗号分隔，实时提示输入格式）；“预览匹配文件”按钮，点击后弹出简易表格，显示匹配到的文件名称、原路径、文件大小。

- 技术实现：
       

  - 关键词匹配：将用户输入的字符串按逗号分割为关键词列表，遍历源目录筛选出的MP3文件，判断文件名是否包含列表中任意一个关键词（大小写不敏感）。

  - 前缀匹配：判断文件名是否以用户输入的前缀字符串开头，精准匹配目标文件。

  - 后缀匹配：默认筛选.mp3后缀，用户可输入其他后缀（但仅处理主流音频文件，额外校验后缀合法性）。

- 异常处理：未设置匹配规则时，提示“请至少设置一种匹配规则”，阻止进入下一步；无匹配文件时，提示“无符合规则的MP3文件，请调整匹配规则后重试”。

3.3 步骤3：设置新文件名规则（统一文件命名）

- 界面设计：前缀输入框（提示“输入新文件名前缀，不可包含特殊字符”），核心名规则（保留原核心名、序号命名）单选按钮，序号位数（2位、3位）下拉选择框（默认3位）；“预览新文件名”按钮，点击后显示前5个文件的“原文件名→新文件名”对照列表，清晰呈现命名效果。

- 技术实现：
        

  - 保留原核心名：提取原文件名主体（去除.mp3后缀），拼接用户设置的前缀，后缀仍为.mp3；后续生成的字幕、翻译文件，自动替换后缀为.lrc、.txt。

  - 序号命名：按筛选文件的顺序，生成对应位数的序号（如3位：001、002，通过zfill()方法实现），拼接前缀，形成新文件名，确保命名统一。

- 异常处理：前缀包含/、\、:、*等特殊字符时，实时提示“前缀不可包含特殊字符，请修改”；序号位数仅提供2位、3位选项，禁止用户手动输入，避免格式错误。

3.4 步骤4：整理生成新的目录和文件（文件规整）

- 界面设计：“复制”“移动”操作单选按钮（默认复制），“生成整理目录”按钮，进度条（显示整理进度百分比），进度提示文本（如“正在整理文件，请勿关闭软件”）；整理完成后，提示“目录整理完成，共整理X个音频文件”。

- 技术实现：
        

  - 复制操作：调用shutil.copy2()方法，将匹配的MP3文件从源目录复制至输出根目录，保留文件元数据，源目录文件不变。

  - 移动操作：调用shutil.move()方法，将MP3文件从源目录移动至输出根目录，移动前弹出确认提示框；若文件被占用，自动切换为复制模式，并提示用户。

- 异常处理：文件被占用时，提示“文件正在被占用，已自动切换为复制模式”；输出目录空间不足时，提示“输出目录空间不足，无法完成整理，请清理空间后重试”；个别文件处理失败时，跳过该文件，继续处理下一个，提示“部分文件整理失败，请检查文件权限”。

3.5 步骤5：用户检查目录结构是否合适（确认规整结果）

- 界面设计：“打开输出目录”“重新整理”“确认无误”三个按钮；“打开输出目录”按钮点击后，直接打开输出根目录，供用户手动检查文件命名、目录结构；“重新整理”按钮可跳转至步骤3，允许用户修改新文件名规则，重新执行整理操作；“确认无误”按钮默认灰色，打开目录后激活，点击后进入字幕生成环节。

- 技术实现：调用os.startfile()方法，打开输出根目录；“重新整理”时，先删除输出目录中已整理的音频文件，再重新执行步骤4的整理操作；“确认无误”后，更新临时配置，记录目录确认状态，确保后续步骤可正常执行。

- 异常处理：输出目录无法打开时，提示“无法打开输出目录，请检查目录路径是否有效”。

3.6 步骤6：生成第一个音频的字幕文件（测试用，验证转写效果）

- 界面设计：“生成第一个字幕”按钮、模型选择下拉框（仅提供Whisper base模型，MVP版暂不支持其他模型）、进度提示文本（如“正在生成第一个字幕，预计1-2分钟”）；生成完成后，显示字幕文件路径（如“字幕生成完成，路径：XXX\XXX.lrc”）。

- 技术实现：获取输出目录中第一个音频文件（按文件名升序排序），调用faster-whisper的transcribe()方法，对音频文件进行语音识别（默认识别英文，支持自动语言检测），生成包含时间轴的转录结果，转换为LRC格式（“[时间轴] 文本”格式），保存至输出目录，文件名与音频文件一致，后缀改为.lrc。

- 优化处理：Whisper模型仅在程序启动时加载一次，避免每次生成字幕都重新加载，提升处理速度；生成过程中实时反馈进度，避免用户误以为软件卡顿。

- 异常处理：音频文件损坏时，提示“第一个音频文件损坏，无法生成字幕，请更换音频文件”；字幕生成失败时，提示“字幕生成失败，请检查软件依赖或音频文件”，记录简单错误日志。

3.7 步骤7：用户自行查看（校验字幕效果）

- 界面设计：“打开字幕文件”按钮，点击后用系统默认文本编辑器（如记事本）打开步骤6生成的LRC文件，供用户手动查看字幕的时间轴准确性、识别文本完整性。

- 技术实现：调用os.startfile()方法，关联LRC文件的默认打开程序，确保用户可直接查看、编辑字幕内容；用户查看完成后，无需额外操作，可直接进入下一步。

- 异常处理：字幕文件不存在时，提示“字幕文件未生成，请先生成第一个字幕”；字幕文件无法打开时，提示“无法打开字幕文件，请检查文件是否存在或损坏”。

3.8 步骤8：批量生成字幕（核心功能，高效转写）

- 界面设计：“批量生成字幕”按钮、进度条（显示当前处理进度、已处理/总文件数）、简易日志区（实时显示每一个文件的处理状态，如“正在处理：XXX.mp3”“处理成功：XXX.lrc”“处理失败：XXX.mp3”）；支持点击“暂停”“继续”按钮，控制批量处理流程。

- 技术实现：沿用步骤6的Whisper base模型，遍历输出目录中所有整理后的MP3文件，按顺序生成对应的LRC字幕；调用os.path.exists()方法，判断输出目录中是否已存在对应字幕文件，若存在则跳过，避免重复生成；批量处理过程中，实时更新进度条和日志，确保用户了解处理状态。

- 任务控制：点击“暂停”后，停止当前文件处理，按钮变为“继续”；点击“继续”后，恢复处理；点击“终止”后，停止所有文件处理，保留已生成的字幕文件，提示“批量字幕生成已终止”。

- 异常处理：部分文件处理失败时，提示“部分文件字幕生成失败，请查看日志了解详情”，跳过失败文件，继续处理下一个；批量处理中断（如软件意外关闭），重启软件后可读取临时配置，继续处理未完成的文件。

3.9 步骤9：翻译第一个字幕（选择本地/AI，AI需配置Key）

- 界面设计：“本地翻译”“AI翻译”单选按钮（默认本地翻译）；AI翻译选项下方设置API Key输入框（带隐藏/显示功能，输入时默认隐藏字符），提示“请输入DeepSeek AI翻译API Key”；“翻译第一个字幕”按钮、进度提示文本，生成完成后显示翻译文件路径。

- 技术实现：


  - 本地翻译：调用CSANMT本地翻译模型，读取步骤6生成的LRC文件中的文本内容，逐句翻译为中文，生成“英文原文→中文翻译”的对照格式，保存为TXT文件，存放在输出目录。

  - AI翻译：调用requests.post()方法，对接DeepSeek AI翻译API，传入API Key、英文文本（提取自LRC文件），设置翻译方向为“en→zh”，获取翻译结果后，按对照格式保存为TXT文件。

- API Key处理：用户输入API Key后，通过base64加密保存至本地临时配置文件，不写入日志、不联网上传，读取时解密，保障信息安全。

- 异常处理：选择AI翻译但未配置API Key时，提示“请先配置DeepSeek AI翻译API Key”；API调用失败（如Key错误、额度不足）时，提示“AI翻译失败，请检查API Key或网络，可切换为本地翻译”；字幕文件不存在时，提示“字幕文件未生成，请先生成第一个字幕”。

3.10 步骤10：用户自行查看（校验翻译效果）

- 界面设计：“打开翻译文件”按钮，点击后用系统默认文本编辑器打开步骤9生成的TXT翻译文件，供用户手动查看英文与中文的对应准确性、翻译流畅度，校验翻译效果。

- 技术实现：调用os.startfile()方法，关联TXT文件的默认打开程序，用户可直接查看、编辑翻译内容；若用户对翻译效果不满意，可点击“重新翻译”按钮，返回步骤9重新执行翻译操作（可切换翻译方式）。

- 异常处理：翻译文件不存在时，提示“翻译文件未生成，请先翻译第一个字幕”；翻译文件无法打开时，提示“无法打开翻译文件，请检查文件是否存在或损坏”。

3.11 步骤11：批量翻译（核心功能，高效翻译）

- 界面设计：“批量翻译”按钮、进度条（显示当前处理进度、已处理/总文件数）、简易日志区（实时显示每一个文件的翻译状态）；支持“暂停”“继续”“终止”操作，与批量生成字幕操作逻辑一致；沿用步骤9选择的翻译方式，无需用户重新选择。

- 技术实现：遍历输出目录中所有生成的LRC字幕文件，按步骤9的翻译逻辑，批量生成对应的TXT翻译文件；调用os.path.exists()方法，判断是否已存在对应翻译文件，若存在则跳过，避免重复生成；翻译文件命名格式为“音频新文件名_中文翻译.txt”，确保与音频、字幕文件对应。

- 特殊处理：AI翻译时检测网络连接，若断网则自动切换为本地翻译，提示“网络中断，已切换为本地翻译”，继续批量处理；本地翻译无需联网，确保离线状态下可正常使用。

- 异常处理：部分文件翻译失败时，提示“部分文件翻译失败，请查看日志了解详情”；API Key失效（AI翻译时），提示“API Key失效，请重新配置”，引导用户重新输入或切换为本地翻译；批量处理中断后，重启软件可继续处理未完成的文件。

3.12 步骤12：完成（流程收尾）

- 界面设计：显示全流程处理汇总（如“处理完成：音频X个、字幕X个（成功X个/失败X个）、翻译X个（成功X个/失败X个）”）；“打开输出目录”“重新开始”两个核心按钮，用户可打开目录查看最终所有文件，或重新开始整个流程。

- 技术实现：统计批量字幕生成、批量翻译的成功/失败数量，清晰展示在界面上；“打开输出目录”按钮调用os.startfile()方法，打开输出根目录；“重新开始”按钮点击后，清空本地临时配置文件中的所有状态（目录、规则、翻译方式、API Key等），关闭当前界面，返回步骤1，允许用户重新操作。

- 异常处理：日志导出失败时，提示“日志导出失败，请检查输出目录权限”；输出目录无法打开时，提示“无法打开输出目录，请检查目录路径是否有效”。

四、非功能需求（MVP精简版，补充关键细节）

- 性能要求：适配普通笔记本（i5处理器+8GB内存），单3分钟音频字幕转写≤90秒，单字幕翻译≤30秒；批量处理时，CPU占用≤70%，内存占用≤2GB，不出现卡顿、崩溃现象。

- 兼容性要求：仅适配Windows 10/11（64位）系统，无需适配多分辨率、多系统主题，确保界面无错乱、按钮无遮挡，核心功能正常运行。

- 稳定性要求：核心操作（文件操作、模型调用、API调用）均添加try-except异常捕获，遇到异常时友好提示错误原因，不崩溃、不闪退；失败文件自动跳过，记录简单错误日志，便于后期排查；临时配置实时保存，软件重启后可恢复之前的操作状态。

- 易用性要求：界面极简、布局清晰，按12步流程分区展示，当前步骤高亮显示，未完成步骤灰色不可点击；每步操作不超过2次点击，提示信息简洁明了，包含问题原因和解决方法，非技术用户可快速上手。

- 安全性要求：用户配置的DeepSeek API Key采用base64加密存储在本地，不联网上传、不泄露；软件仅申请读取源目录、写入输出目录的必要权限，不申请其他无关权限，不收集用户个人信息和文件内容，保障用户隐私和文件安全。

五、开发约束与部署要求（MVP版，明确落地标准）

- 依赖约束：Whisper base模型、CSANMT本地翻译模型均内置打包，无需用户额外下载、配置；AI翻译仅对接DeepSeek AI翻译API，不支持其他第三方API，降低开发复杂度。

- 打包要求：使用PyInstaller打包为单一exe安装包，配置打包参数，精简依赖组件，去除不必要的冗余文件，将安装包体积控制在300MB以内；支持一键安装，安装过程中可选择安装路径；卸载时自动删除安装目录下的所有文件，不残留配置文件和缓存文件。

- 代码要求：遵循Python PEP 8代码规范，变量、函数命名统一，添加必要的核心注释（无需冗余注释），聚焦12步核心逻辑落地，便于后期快速迭代和bug修复。

- 权限要求：软件启动时，自动校验当前用户对常用目录（桌面、文档、自定义目录）的读写权限，无权限则提示用户“以管理员身份运行软件”，避免文件操作失败。

六、测试要求（MVP精简版，保障功能可用）

- 测试范围：覆盖12步核心流程的全场景测试，重点测试文件整理、字幕转写、翻译三大核心功能；测试异常场景（如目录无权限、文件损坏、API Key错误），确保异常处理有效，软件不崩溃。

- 测试环境：
        

  - 硬件环境：普通笔记本（i5处理器+8GB内存+100GB以上空闲磁盘）、台式机（i7处理器+16GB内存）。

  - 软件环境：Windows 10（64位）、Windows 11（64位）系统，默认文本编辑器（记事本）。

- 测试标准：12步流程连贯可执行，无卡顿、无崩溃；音频整理后命名规范，字幕文件时间轴准确、识别完整，翻译文件对照清晰、无明显错误；异常场景提示准确，失败文件可正常跳过，软件可正常继续运行。
